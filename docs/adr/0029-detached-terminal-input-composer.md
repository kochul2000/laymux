# 0029. Terminal 분리 입력 컴포저 — PC와 Remote 공통 surface 모델

- Status: Accepted
- Date: 2026-07-13

## Context

PC `TerminalView`와 Focused Remote UI는 모두 xterm.js의 helper textarea가 키보드·IME 입력을 받고, `terminal.onData`가 생성한 바이트를 하나의 PTY에 보내는 직접 입력 모델을 쓴다. 이 모델은 실제 터미널과 가장 가까운 호환성을 제공하지만, 긴 자연어 prompt나 다중행 텍스트를 전송 전에 검토·수정하기 어렵다. 특히 모바일 Remote에서는 소프트 키보드가 viewport와 겹치고 조합 중인 문자열을 다루기 불편하다.

xterm.js는 “사용자가 편집하는 입력 caret”과 “PTY 애플리케이션이 마지막으로 이동한 터미널 커서”를 구분하지 않는다. Codex·Claude Code 같은 TUI가 footer를 반복해서 다시 그리면 단일 애플리케이션 커서는 입력 위치와 리페인트 위치 사이를 이동한다. PC `TerminalView`는 OSC 133/633, 프레임 밖 DECTCEM cursor park, save/restore, DEC 2026, `onWriteParsed`를 조합한 shadow cursor와 별도 IME composition preview로 이를 완화한다([ADR-0008](0008-shell-cursor-shadow-cursor.md), [ADR-0011](0011-dectcem-cursor-park-fifth-layer.md)). 그러나 커서 추적은 이미 PTY에 들어간 입력의 표시를 안정화할 뿐, 전송 전 텍스트 편집기를 제공하지는 않는다. 같은 로직을 Remote 정적 페이지에 복제해도 모바일 입력 불편은 남는다.

Remote xterm의 parser mode도 PTY 전역 상태의 신뢰 가능한 원본이 아니다. 최초 attach와 output reconnect는 xterm을 reset한 뒤 output ring의 최근 64 KiB만 재생한다. TUI가 그보다 앞서 bracketed paste를 `CSI ? 2004 h`로 켰다면 최근 snapshot에는 mode 전이가 없고, 새 xterm의 `terminal.modes.bracketedPasteMode`는 실제 애플리케이션 상태와 달리 `false`가 된다. 이를 그대로 믿고 다중행 text를 보내면 각 CR이 별도 제출로 해석될 수 있다.

controller owner도 frontend status cache만으로 강제할 수 없다. Remote claim 직후 PC가 `remote-control-changed`를 처리하기 전에 Tauri `write_to_terminal`을 호출하면, 현재 backend command는 terminal id와 data만 확인하고 PTY에 기록한다. owner 확인과 실제 write가 하나의 backend 임계 구역에서 선형화되지 않으면 UI 차단은 안전 경계가 아니다.

현재 `RemoteControlState`의 lease와 `/remote/v1/session/claim`, `/remote/v1/session/release`는 terminal id가 없는 프로세스 전역 계약이다. 따라서 terminal별 write gate만으로는 owner 전환을 직렬화할 수 없다. terminal A의 Local write가 검증을 마친 뒤 전역 claim이 성공하고 그 write가 나중에 PTY에 반영되면, claim 완료 뒤에는 stale PC 작업이 없다는 controller 계약을 위반한다.

composer commit도 비동기 작업이다. 전송 중 같은 textarea를 편집하거나 Send를 다시 누를 수 있는데, 응답이 왔다는 이유만으로 현재 초안을 비우면 제출 뒤 추가한 텍스트를 잃는다. 반대로 중복 action을 허용하면 같은 snapshot이 두 번 기록될 수 있다. 제출 당시 초안과 완료 시점의 초안을 구분하는 client-side revision과 in-flight 상태가 필요하다.

반대로 Warp처럼 셸 line editor의 상태를 laymux가 소유하려면 shell integration과 애플리케이션별 의미 해석이 필요하다. PTY 화면에서 현재 입력 문자열을 역추론해 별도 editor와 양방향 동기화하는 것은 터미널 프로토콜의 책임 경계를 넘으며 임의의 TUI와 호환되지 않는다.

따라서 PC와 Remote 모두 기존 직접 입력을 보존하면서, 아직 PTY에 보내지 않은 텍스트만 네이티브 editor에서 편집하는 선택적 분리 모드가 필요하다. 이 모드는 하나의 PTY와 controller owner 경계를 그대로 사용해야 하며, 각 client의 초안이나 renderer 상태를 새로운 PTY 진실원으로 만들지 않아야 한다.

## Decision

PC `TerminalView`와 Focused Remote UI는 terminal surface마다 다음 두 입력 방식을 제공한다.

1. **Direct mode (`direct`)**: 기존 동작이다. xterm helper textarea가 키보드·IME 입력을 받고 `terminal.onData`를 현재 surface의 write adapter로 전달한다. PC의 5-layer shadow cursor와 composition preview도 이 모드에서 그대로 유지한다. 전체화면 TUI, 마우스 추적, 키 단위 제어가 필요할 때 사용한다.
2. **Detached composer mode (`composer`)**: terminal output viewport와 분리된 네이티브 `textarea`에서 전송 전 텍스트를 편집한다. 이 textarea의 value, selection/caret, IME composition만 미전송 입력의 진실원이다. xterm은 PTY output을 계속 파싱·렌더링하고 terminal mode와 geometry를 추적하지만 텍스트 입력 focus를 갖지 않는다. PC는 xterm native cursor뿐 아니라 shadow overlay caret과 composition preview도 숨기며, Remote는 xterm 애플리케이션 커서를 숨긴다.

모드와 상태 소유권은 다음 규칙을 따른다.

- 각 terminal surface의 현재 모드와 초안은 terminal id별 client 메모리 상태다. 명시적으로 고른 기본 모드 선호만 surface별 `localStorage`에 저장한다: PC는 `laymux.desktop.inputMode`, Remote는 `laymux.remote.inputMode`를 사용한다. PC의 최초 기본값은 항상 `direct`이며, Remote는 저장된 선호가 없을 때 coarse pointer에서 `composer`, fine pointer에서 `direct`를 사용한다.
- composer 초안은 서버, `settings.json`, `localStorage`에 저장하지 않는다. terminal을 바꾸거나 Remote lease를 일시적으로 잃어도 같은 client runtime 동안 terminal id별 초안을 보존하지만, WebView/페이지 reload 또는 앱 종료 뒤에는 복구하지 않는다. 이는 명령·토큰 같은 민감한 미전송 문자열을 영속화하지 않기 위한 경계다.
- PC와 Remote는 같은 `ComposerDraftState { text, revision, inFlight }` 전이 계약을 사용한다. terminal별 `revision`은 text가 바뀔 때마다 증가한다. Insert/Send는 `inFlight`가 없을 때만 local submission token과 `{ terminalId, revision, text, submit }` snapshot을 원자적으로 기록하고 요청을 한 번 시작한다. 같은 terminal의 Insert/Send action은 해당 token이 settle될 때까지 비활성화해 키 반복·더블 클릭·재호출로 같은 snapshot을 중복 전송하지 않는다. textarea 편집은 계속 허용하고 변경 시 revision을 올린다. 성공 응답은 token이 현재 `inFlight`와 일치하고 현재 revision과 text가 제출 snapshot과 모두 같을 때만 초안을 비운다. 제출 뒤 text가 바뀌었으면 새 초안을 그대로 보존하며 자동 diff·병합을 하지 않는다. 실패나 결과가 불명확한 응답도 현재 초안을 보존한다. terminal 전환 뒤 도착한 응답은 요청에 캡처된 terminal의 상태에만 적용하고, 이미 교체된 token의 stale 응답은 무시한다. 이 상태와 token도 client runtime에만 존재한다.
- bracketed-paste 여부는 xterm surface가 아니라 **Rust PTY output callback이 소유하는 PTY 전역 protocol state**다. backend는 chunk 경계를 보존하는 streaming DEC private-mode tracker로 전체 PTY output의 `CSI ? 2004 h/l`과 terminal reset을 처리하고, terminal별 mutex gate 안에 `TerminalProtocolState { bracketed_paste, revision }`를 유지한다. tracker는 여러 DEC parameter와 chunk 중간 분할을 처리하며 recent output ring이 오래된 bytes를 버려도 상태를 잃지 않는다. 이 per-terminal gate는 바깥 table lock을 놓은 뒤 획득한다. human control 경로에서는 아래의 owner permit 등록을 완전히 끝내고 owner gate를 놓은 뒤, 다른 AppState lock보다 먼저 잡는다. output callback과 attach 경로는 owner gate나 control worker 없이 protocol-state gate→`output_buffers` 순서만 사용한다. 이 순서를 `state.rs`에 추가한다.
- 모든 attach/reattach는 output snapshot과 같은 시점의 `TerminalAttachState { version, snapshotStartSeq, snapshotSeq, modes: { bracketedPaste } }`와 snapshot bytes를 한 묶음으로 받는다. sequence는 terminal이 생성된 뒤 Rust callback이 수신한 raw output byte offset이며 `[snapshotStartSeq, snapshotSeq)`가 snapshot의 정확한 범위다. PTY callback과 attach snapshot은 모두 terminal protocol-state gate→`output_buffers` 순서로 처리하므로 mode와 sequence 범위가 같은 output prefix를 나타낸다. 모든 live delta도 `TerminalOutputDelta { seqStart, seqEnd, data }` 범위를 가지며 중복은 잘라 내고 gap은 임의로 메우지 않고 새 attach를 시작한다. Direct Remote와 Cloud host 모두 같은 gate 구간에서 output consumer를 먼저 등록하고 state/snapshot을 캡처해 이후 output을 consumer buffer에 쌓는 server-side subscription barrier를 사용한다.
- Direct와 Cloud browser WebSocket은 공통 `TerminalOutputFrameHeaderV1 { type: "terminal.output", version: 1, phase: "snapshot" | "delta", seqStart, seqEnd, byteLength, state? }` text header 바로 뒤에 정확히 한 개의 binary bytes frame을 보내며 두 frame 사이에 다른 output frame을 끼우지 않는다. snapshot header에만 `state`가 필수이고 `seqStart == state.snapshotStartSeq`, `seqEnd == state.snapshotSeq`여야 한다. 빈 snapshot도 길이 0인 binary frame을 보내지만, 새 bytes가 없는 idle poll은 delta header 자체를 보내지 않는다. 첫 frame은 snapshot이고 이후 delta의 sequence는 단조 증가해야 한다. `byteLength == seqEnd - seqStart`와 실제 binary 길이가 모두 일치하지 않거나 phase/order/state 범위가 어긋나면 client는 stream을 버리고 재attach한다. Direct server는 이 pair를 그대로 보낸다. Cloud host는 `stream.open{kind:"websocket.accept", outputProtocol:"laymux-terminal-output.v1", attachState}`를 먼저 보내고, 각 `stream.data` payload를 `{ encoding:"base64", data, output:{ version:1, phase, seqStart, seqEnd, byteLength } }`로 확장한다. relay는 accept의 `attachState`를 snapshot header의 `state`로 결합하고, 각 `stream.data`를 공통 text-header→decoded-binary pair로 순서 보존 변환한다. HTTP stream과 다른 WebSocket의 `stream.data` 형식은 바꾸지 않는다([ADR-0024](0024-cloud-native-wss-tunnel.md)).
- PC WebView는 별도 attach-state query를 사용하지 않는다. 먼저 versioned `terminal-output-v2-{id}` listener를 등록해 delta를 임시 buffer에 모은 뒤, 새 Tauri `attach_terminal_output` RPC 한 번으로 `{ state, snapshot }`을 원자적으로 받는다. client는 `seqEnd <= snapshotSeq`인 buffered delta를 버리고 `seqStart < snapshotSeq < seqEnd`인 delta는 `snapshotSeq`부터의 suffix만 남긴 뒤 sequence 순으로 적용한다. 다음 expected sequence보다 큰 `seqStart`는 gap이다. 이 listener-before-RPC 절차에서는 listener 등록 이전 output은 snapshot에 포함되고 등록 이후 output은 snapshot 또는 buffered delta 중 적어도 하나에 포함되므로 query/listener 사이의 유실과 역전이 없다. 기존 unsequenced `terminal-output-{id}` event는 이 surface의 attach 경로에 섞지 않는다.
- PC의 `SerializeAddon` 영속 output cache도 같은 attach coordinator만 xterm에 쓸 수 있다. cache load와 PTY create/attach RPC는 latency를 줄이기 위해 병렬 실행할 수 있지만, listener 등록 뒤 live delta를 buffer한 상태에서 실제 parser write는 반드시 **`reset → cache(있을 때) → session-restored separator → live snapshot → synthetic final mode → sequenced delta`** 순서로 await한다. cache와 snapshot 안의 `CSI ? 2004 h/l`을 따로 신뢰하지 않고 둘을 모두 parse한 마지막에 snapshot 끝 시점의 synthetic mode를 적용한다. 각 mount/reattach는 `attachEpoch`을 가지며 늦게 끝난 cache/RPC promise는 현재 epoch와 일치할 때만 쓰므로, 이전 cache가 synthetic mode나 새 attach 뒤에 도착할 수 없다. attach 중 오류나 재시작은 이전 epoch의 buffered delta와 pending write를 버리고 `reset`부터 다시 시작한다.
- client는 attach state를 받기 전 mode를 `false`로 추정하지 않는다. 위 ordered pipeline에서 snapshot write가 parse를 마칠 때까지 기다린 다음, snapshot 끝 시점의 backend state에 맞는 synthetic `CSI ? 2004 h/l`을 **xterm parser에만** 적용하고 `snapshotSeq` 이후 buffered/live delta를 재생한다. synthetic mode를 cache/snapshot보다 먼저 적용하면 그 안의 과거 mode 전이가 authoritative final state를 다시 덮을 수 있으므로 금지한다. live delta는 backend tracker와 각 xterm parser가 모두 처리한다. attach metadata가 없거나 version을 지원하지 않거나 sequence gap을 복구하지 못하면 composer Insert/Send와 direct clipboard paste를 fail-closed로 비활성화하며 raw 다중행 paste를 보내지 않는다.
- composer는 output viewport 아래에 배치해 terminal 내용을 덮지 않는다. composer 높이가 바뀌면 해당 xterm surface의 가용 geometry도 바뀐다. 현재 controller owner인 surface만 최종 `cols/rows`를 PTY에 보내며, owner가 아닌 PC surface는 기존 remote-return dirty/reflow 경로로 최신 geometry를 나중에 동기화한다([ADR-0015](0015-remote-terminal-state-ownership.md)).
- composer가 활성일 때 terminal tap·선택·스크롤은 xterm helper textarea에 사용자 입력 focus를 주지 않는다. 다만 xterm의 `terminal.onData` 전체를 모드로 차단하지 않는다. focus reporting이나 terminal query 응답처럼 xterm이 생성하는 protocol data도 같은 채널을 통과할 수 있으므로, 사용자 키 입력은 focus 경계에서 막고 필요한 protocol reply는 backend owner gate를 통과해 전달한다.
- 공통 action은 **Insert**와 **Send**다. Insert는 초안을 현재 PTY 입력 위치에 붙여 넣되 Enter를 추가하지 않는다. Send는 같은 payload 뒤에 단독 CR(`\r`)을 추가한다. client는 `{ text, submit }` intent만 보내고, Rust가 줄바꿈을 CR로 정규화한 뒤 authoritative `TerminalProtocolState.bracketed_paste`가 참이면 텍스트 부분만 `CSI 200~`/`CSI 201~`로 감싼다. CR 제출은 bracketed-paste 종료 뒤에 둔다. direct clipboard paste도 같은 structured input 경로를 사용해 stale xterm mode로 raw paste를 만들지 않는다. PC의 key gesture는 키바인딩 registry action으로만 연결하고 컴포넌트에 조합을 하드코딩하지 않는다. Remote에서는 IME composition 중 Enter를 제출로 해석하지 않으며 일반 Enter는 Send, Shift+Enter는 줄바꿈으로 처리한다.
- 한 composer commit은 backend가 준비한 text와 선택적 CR을 합친 **하나의 terminal input job**이다. PC는 새 Tauri `write_terminal_input`, Remote는 새 lease 보호 `/remote/v1/terminals/{id}/input` structured endpoint를 사용한다. 기존 raw Tauri `write_to_terminal`과 Remote `/write`는 direct key·protocol reply·soft-key 용도로 유지하되 아래 owner permit 경로를 반드시 공유한다. 같은 terminal의 per-terminal control worker가 input/resize job을 FIFO로 직렬화하고 한 input job의 physical chunk 사이에 다른 job을 끼우지 않는다. 초안 정리는 위 revision/token 규칙에만 따른다.
- Rust는 human surface의 raw/structured write와 resize를 단일 owner-checked 내부 경로로 모은다. origin은 `Local` 또는 `Remote { lease_id }`다. `AppState`에는 프로세스 전역 **`remote_owner_gate: RwLock<RemoteOwnerState>`**를 두고, `RemoteOwnerState`가 lease, 단조 증가 `owner_epoch`, 전환 상태, claim reservation, 등록된 operation permit을 함께 소유한다. 이 gate는 권한 검증·permit 등록/완료·epoch 전환 동안만 짧게 잡고 terminal lock이나 PTY I/O를 잡은 채 획득하지 않는다. write/resize는 write guard 아래에서 현재 owner와 전환 상태를 검증하고 `{ owner_epoch, operation_id, deadline, cancel_token }` permit을 등록한 뒤 guard를 놓는다. 전환 중에는 양쪽 owner의 새 permit을 모두 fail-closed한다. 이후 protocol mode 캡처와 queue 대기 중에도 epoch/token을 검사하며, worker는 physical operation 직전과 직후에 다시 검사한다. permit deadline은 등록 시점부터 흐르고 request task가 enqueue 전에 멈춰도 watchdog/다음 owner operation이 만료 permit을 `Cancelling`으로 옮긴다. permit은 worker의 completion/cancellation acknowledgement 또는 독립 lifecycle handle의 PTY 종료 acknowledgement 뒤에만 제거하며, 그 전에는 claim/owner publish barrier에 계속 포함한다. frontend status event는 UX 반영용일 뿐 권한 경계가 아니며 Automation/MCP의 명시적 automation writer는 별도 신뢰 경계로 유지한다.
- `RemoteControlLease`는 `last_heartbeat + 현재 설정값`으로 유효성을 매번 재계산하지 않고, claim 또는 성공한 heartbeat 시점에 그때의 effective timeout으로 정한 절대 monotonic `deadline`과 `Active | Expiring` phase를 저장한다. `now >= deadline`이 한 번 관찰되면 같은 lease id는 비가역적으로 `Expiring`이며 heartbeat가 갱신할 수 없다. timeout 설정 변경은 다음 성공한 claim/heartbeat가 만드는 deadline에만 반영한다. status·owner 검증·claim 중 어느 경로가 먼저 만료를 발견하든 owner write guard 아래에서 phase를 바꾸고 epoch를 증가시켜 해당 epoch의 queued/in-flight Remote permit을 취소한다. cancellation barrier가 끝나기 전에는 기존 `active`를 true로 유지하면서 `transitioning: true`를 함께 보내 구버전 PC도 Local owner로 오인하지 않게 하고, 양쪽 write/resize를 fail-closed한다. barrier 뒤 lease를 제거하고 한 번만 Local 상태 event를 emit하므로 timeout 증가로 옛 lease가 부활할 수 없다.
- `/claim`은 먼저 owner read guard로 active unexpired lease와 reclaim lockout을 확인하고 해당하면 write/exclusive waiter가 되지 않은 채 즉시 `409`로 거절한다. 전환 가능성이 보일 때만 write guard를 얻어 만료·lease·epoch를 다시 검증한다. lease가 없더라도 기존 Local operation permit이 있으면 claim은 guard나 I/O 종료를 기다리지 않고 `{ code:"input_busy", claimReservationId, retryAfterMs, reservationTtlMs }` `409`를 반환한다. 이때 `RemoteOwnerState`에 예측 불가능한 one-shot token의 hash, 관찰한 owner epoch, absolute monotonic expiry를 가진 `ClaimReservation`을 설치하고 raw token은 이 인증된 응답에서만 반환한다. 후속 claim은 optional `claimReservationId`로 token을 제시한다. reservation은 기존 permit 완료 뒤에도 지우지 않고 같은 token의 재시도가 성공적으로 소비하거나 expiry/local reclaim/Remote access disable이 명시적으로 취소할 때까지 새 Local permit과 token 없는/다른 token의 claim을 거절한다. token 재시도는 write guard에서 constant-time token hash·epoch·expiry와 permit 0을 다시 확인하고 reservation 소비·epoch 증가·lease 설치를 원자적으로 수행하며, permit이 남아 있으면 같은 token의 남은 TTL로 다시 `input_busy`를 반환한다. 따라서 active controller의 정상 heartbeat를 선점하지 않으면서도 permit 완료 직후 Local key job이 재시도 claim을 앞지를 수 없다. 성공한 claim 뒤에는 예전 Local job이 남지 않으며, reservation이 만료되면 Local 입력을 자동 재개한다. release, local reclaim, Remote access disable과 만료 cleanup은 write guard에서 먼저 transition을 기록하고 epoch를 올려 새 old-owner job 등록을 차단한 뒤 취소 대상 permit을 떼어 내며, guard 밖에서 아래 bounded cancellation barrier를 수행한다.
- 기존 `PtyHandle::write`의 무제한 `write_all`/`flush`는 human input 경로에서 사용하지 않는다. per-terminal control worker는 플랫폼별 `InterruptiblePtyInput`과 bounded resize adapter를 소유하고, 각 physical operation에 deadline과 cancel handle을 둔다. Unix input은 nonblocking PTY fd와 poll/wakeup으로, Windows ConPTY input은 전용 worker thread의 synchronous pipe write와 그 thread만 겨냥한 `CancelSynchronousIo`로 대기를 깨우며, 한 cancellation이 다음 job의 I/O를 잘못 취소하지 않도록 완료 acknowledgement 뒤에만 worker를 재사용한다. payload 크기·job deadline·cancel grace는 `constants.rs`의 공통 상한으로 제한하고 Tauri/Remote 양쪽에서 동일하게 적용한다. deadline 또는 owner epoch 변경 시 queued suffix를 버리고 진행 중 I/O를 취소한 뒤 acknowledgement를 반환한다. platform adapter가 grace 안에 취소 완료를 증명하지 못하면 worker/writer mutex와 독립된 lifecycle handle로 해당 PTY 전체와 자식 process를 종료하고 terminal을 input-fault 상태로 격리한다. old epoch가 더 기록될 수 없다는 종료 acknowledgement 뒤에만 owner 전환을 publish하며, 이 fail-closed 처리는 terminal session 손상을 알리고 payload를 자동 재전송하지 않는다. 여러 terminal의 취소/종료 barrier는 병렬 fan-out하고 하나의 전역 cancel-grace 상한을 공유한다.
- owner 전환은 global gate를 PTY I/O 동안 보유하지 않는다. 강제 local reclaim과 Remote access disable은 epoch 변경 뒤 bounded cancellation barrier만 기다리며, stdin을 읽지 않는 자식의 무제한 `write_all` 완료를 기다리지 않는다. 전환이 성공해 Local 상태를 publish한 뒤에는 old epoch worker가 어떤 byte도 새로 기록할 수 없다. 이미 cancellation 전에 kernel이 수락한 prefix는 되돌릴 수 없으므로 write 결과는 성공·실패·취소 모두 전송량을 확정하지 않는 ambiguous outcome이 될 수 있다. 특히 Remote HTTP 결과를 받지 못했거나 reclaim에 의해 취소되면 사용자가 output을 확인한 뒤 명시적으로 다시 보내야 한다([ADR-0027](0027-remote-connection-graceful-recovery.md)). 새 request id나 exactly-once 서버 계약은 추가하지 않는다.
- structured 요청은 permit 등록 뒤 terminal protocol-state gate에서 현재 mode와 revision을 캡처해 payload를 인코딩하고 gate를 놓은 다음 control worker에 enqueue한다. mode 캡처·인코딩이 해당 요청과 mode 전이 사이의 선형화 지점이며 이후 mode 전이는 다음 요청부터 적용한다. raw write는 이 gate를 건너뛴다. output callback은 global owner gate나 control worker를 거치지 않고 protocol-state gate→`output_buffers`만 사용하므로 input backpressure 중에도 stdout을 drain한다. 이 경로의 lock 순서는 owner gate 내부의 짧은 permit 등록을 완전히 끝낸 뒤 terminal protocol-state gate→필요한 AppState table lock이며, 실제 queue wait/PTY I/O 중에는 owner/protocol/AppState lock을 하나도 보유하지 않는다.
- PC에서 composer commit은 direct typing과 같은 “사용자가 응답함” action으로 처리해 기존 notification dismiss 정책을 재사용한다. Remote의 소프트 키 툴바와 `Ctrl+C`는 두 모드 모두에서 기존 PTY 직접 입력 경로를 사용하며 composer 문자열을 편집하지 않는다([ADR-0028](0028-remote-soft-key-toolbar.md)).
- active Remote lease가 있으면 PC composer 초안은 보존하되 기존 local input overlay 아래에서 편집·Insert·Send를 비활성화하고 PTY resize도 보내지 않는다. lease가 끝나면 PC는 기존 명시적 reflow/resize 복구 뒤 편집과 전송을 재개할 수 있다. 어느 surface도 controller owner가 아닐 때 PTY write/resize를 보내지 않는다.

분리 모드는 PC와 Remote가 공유하는 입력 보조 개념이지만 셸/TUI line editor의 복제본은 아니다. 현재 PTY 입력 줄을 composer로 가져오거나, composer selection을 애플리케이션 커서와 동기화하거나, 이미 전송된 내용을 양방향 편집하지 않는다. Direct mode의 기존 cursor·IME·TUI 호환 동작도 변경하지 않는다.

## Implementation Plan

1. **PTY protocol-state tracker 테스트를 먼저 추가한다.** `CSI ? 2004 h/l`, 여러 parameter, chunk 경계 분할, ring eviction, reset을 순수 streaming parser test로 고정한다. PTY callback과 attach가 terminal protocol-state gate→output ring 순서를 공유해 bytes·`snapshotSeq`·mode를 같은 output prefix에서 캡처하는지 검증한다.
2. **attach protocol 테스트를 먼저 추가한다.** Direct WebSocket의 common V1 header→binary snapshot/delta pair와 reconnect 복구를 검증한다. Cloud tunnel은 `websocket.accept`의 protocol/state, metadata가 붙은 `stream.data`, relay의 동일 browser pair 변환을 wire fixture로 고정하고 empty/mismatched length와 Cloud reconnect overlap/gap을 포함한다. PC는 listener를 먼저 등록하고 RPC 및 cache 응답을 서로 다른 순서로 의도적으로 늦춰 event 중복 suffix가 잘리고 gap이 재attach를 일으키는지 검증한다. cache와 snapshot 양쪽에 final state와 반대인 과거 `CSI ? 2004 h/l`이 있어도 `reset→cache→snapshot→synthetic→delta`가 최종 상태를 복구하고, stale attach epoch의 cache가 뒤늦게 쓰이지 않아야 한다. metadata 누락·지원하지 않는 version·복구 불가능한 gap에서 paste/composer가 fail-closed인지 Playwright로 확인한다.
3. **backend owner epoch와 lease 전이 테스트를 먼저 추가한다.** Remote claim 전/후 Local write, 잘못된/정상 lease의 Remote write, claim과 Local write 경쟁, local/remote resize를 Rust test로 검증한다. active lease와 긴 Remote input job이 있는 동안 다른 peer의 claim이 shared precheck에서 즉시 거절되고 기존 heartbeat가 deadline을 갱신하는지 검증한다. 30초 deadline이 이미 지난 뒤 설정을 60초로 늘려도 같은 lease의 status/heartbeat가 되살아나지 않고, expiry cleanup의 cancellation barrier 전에는 Local owner가 노출되지 않아야 한다. Local permit이 있는 claim은 reservation id를 반환하고, permit 완료 뒤 새 Local job은 계속 차단되며 같은 token의 재시도만 reservation을 원자적으로 소비해 성공해야 한다. 누락·오류 token과 stale epoch는 거절하고 reservation expiry와 local reclaim은 Local 입력을 복구해야 한다. 성공한 claim 뒤 stale Local 작업이 PTY에 반영되지 않아야 하며, frontend status callback을 의도적으로 늦춘 e2e에서도 PC payload가 Remote owner의 PTY에 들어가지 않아야 한다.
4. **interruptible control worker와 structured input 계약 테스트를 먼저 추가한다.** stdin을 전혀 읽지 않는 fake PTY와 output을 동시에 가득 쓰는 fake child에서 input job deadline, cancellation acknowledgement, output drain을 검증한다. blocking Remote `/write`와 `/input` 각각에 대해 local reclaim과 Remote access disable이 cancel grace 상한 안에 끝나고, 반환 뒤 old epoch byte가 추가되지 않으며 새 Local input이 가능해야 한다. cancellation을 무시하는 fake adapter에서는 독립 lifecycle handle로 PTY 전체를 종료·격리한 뒤에만 Local 상태를 publish하는 fail-closed 경로를 검증한다. authoritative bracketed-paste on/off, 줄바꿈 정규화, Insert/Send CR 위치, 빈 초안, payload 상한, Unicode·한글·다중행 vector를 Rust encoder와 PC/Remote adapter test로 고정한다. mode 캡처 뒤 owner/protocol/AppState lock이 모두 해제된 상태에서 worker가 실행되고, write 도중 생긴 mode 전이가 다음 요청부터 반영되는지도 검증한다. 두 client가 공유할 state-machine vector로 rapid double submit은 한 요청만 만들고, 성공+동일 revision은 clear, 성공+편집된 revision과 실패·취소·ambiguous 결과는 preserve, terminal 전환 뒤 완료와 stale token은 캡처된 terminal에만 조건부 적용되는지 검증한다. direct clipboard paste도 structured 경로를 사용하고 raw key/protocol reply도 owner permit을 등록하는지 검증한다.
5. **PC `TerminalView`와 재사용 UI를 TDD로 구현한다.** direct 기본값, terminal별 mode/draft 격리, native·shadow·composition caret 비표시, composer focus, in-flight action 차단과 revision 조건부 clear, notification dismiss, 단일 attach coordinator의 cache/snapshot/delta ordering, Remote lease 중 backend write/resize 거절과 lease 반환 후 geometry 복구를 검증한 뒤 `components/ui/` composer와 키바인딩 registry action을 연결한다.
6. **Remote UI와 input ordering을 TDD로 구현한다.** coarse/fine pointer 기본값, 저장된 선호, terminal별 초안, composition 중 Enter 무시, Direct/Cloud 공통 V1 output frame decoder와 attach state 적용, PC와 같은 in-flight/revision 전이, 단일 structured request, 실패 시 draft 보존과 무자동재시도, soft-key raw 경로를 Playwright와 `page.rs` 계약 test로 검증한다.
7. **living doc과 자율 검증을 완료한다.** `docs/architecture/data-flow.md` §8에 PTY protocol-state/PC composer/owner gate와 persistent cache attach ordering을, `docs/architecture/api-contracts.md` §13.4와 cloud tunnel 계약에 V1 output header/binary pair, relay 변환, claim reservation, structured input endpoint를 반영한다. 관련 Rust/Vitest/Remote Playwright/UI build를 통과시키고 dev 19281에서 PC·Direct Remote·Cloud Remote의 한글 IME, 다중행 paste, reconnect, lease race를 확인한 뒤 모바일·데스크톱 스크린샷을 비교한다. dev 프로세스는 `bash scripts/kill-dev.sh`로만 종료한다.

## Consequences

- PC와 모바일 Remote 모두 전송 전 텍스트를 네이티브 caret, selection, IME로 안정적으로 편집할 수 있다. composer mode에서는 PTY 애플리케이션 커서를 입력 caret으로 사용하지 않으므로 footer repaint가 미전송 입력 caret을 움직이지 않는다.
- PC의 기존 shadow cursor와 composition preview는 제거되지 않는다. Direct mode는 현재 동작을 유지하고, composer mode만 별도의 caret owner를 사용한다.
- 셸과 TUI가 실제로 받은 입력·에코·편집 상태는 계속 PTY 안에만 존재한다. 전송한 뒤 내용을 수정하려면 Direct mode나 terminal control key를 사용해야 한다.
- composer가 차지하는 높이는 surface geometry를 바꾸므로 owner 전환 시 기존 guarded reflow/resize 규칙을 반드시 따른다. Remote 제어 중 PC layout 변화가 shared PTY를 resize하지 않는다.
- PC와 Remote 초안은 서로 동기화되지 않는다. surface-local 미전송 상태를 공유하지 않아 lease 전환과 transport 단절이 draft ownership까지 확장되지 않는다.
- PC와 Remote 모두 전송 중 같은 terminal의 중복 commit을 막는다. 전송 뒤 편집한 revision은 성공 응답에도 지워지지 않으며, 비동기 완료가 다른 terminal의 초안에 영향을 주지 않는다.
- xterm의 mode는 표시용 mirror이고 Rust `TerminalProtocolState`가 bracketed-paste의 권위다. 64 KiB snapshot에서 오래된 mode 전이가 빠지거나 영속 cache에 과거 mode가 들어 있어도 ordered attach pipeline의 마지막 synthetic mode와 backend structured encoder가 안전한 paste를 보장한다.
- human surface의 write/resize 권한은 backend owner epoch와 operation permit으로 강제된다. claim과 동시에 들어온 stale PC 요청은 명시적 owner/busy 오류로 실패하고 draft를 보존한다. active lease에 대한 거절될 claim은 heartbeat 앞의 exclusive waiter가 되지 않는다.
- lease deadline은 생성 시 고정되고 만료 관찰은 sticky transition이므로 설정 변경이나 늦은 heartbeat가 Local로 돌아온 owner를 다시 Remote로 바꾸지 못한다. expiry cleanup 중에는 어느 surface에도 write/resize 권한을 주지 않는다.
- 긴 PTY input은 global owner/protocol/AppState lock을 점유하지 않는다. output callback은 계속 drain하고 강제 reclaim은 cancel grace로 상한이 정해진다. 다만 cancellation 전 kernel이 받은 prefix는 회수할 수 없고, platform cancellation까지 실패하면 권한 경계를 보존하기 위해 해당 terminal을 종료·격리할 수 있으므로 client는 draft를 보존하고 결과를 ambiguous로 표시한다.
- 새 Tauri structured-input/atomic-attach 계약, versioned sequenced output event, Remote `/input` endpoint, one-shot claim reservation, Direct/Cloud 공통 output header/binary pair와 Cloud `websocket.accept`/`stream.data` metadata가 추가된다. settings schema는 바뀌지 않는다. Cloud relay는 host metadata를 공통 browser frame pair로 변환해야 하며, 구버전 relay에서는 paste/composer를 fail-closed로 제한한다.
- 현재 Remote entry가 정적 HTML인 동안 PC React component와 Remote composer 구현은 별도다. 공통 test vector와 wire 계약으로 drift를 막고, ADR-0013의 React remote client adapter가 완성되면 UI component를 합친다.
