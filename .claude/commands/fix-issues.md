# Parallel Issue Fix (Worktree Sub-agents)

사용자가 전달한 이슈 번호들을 각각 독립된 워크트리 서브에이전트에서 병렬로 수정한다.

## 절차

### 1. 이슈 번호 파싱

`$ARGUMENTS`에서 이슈 번호를 추출한다. `#` 접두사는 있든 없든 허용.
예: `#10 #12 15` → `[10, 12, 15]`

인자가 없으면 자동 분석 모드로 진입한다. 한 번의 호출로 모든 이슈를 가져온다:

```bash
gh issue list --state open --limit 20 --json number,title,labels,body
```

모든 이슈를 분석하여 **병렬 실행 가능한 그룹**을 자동으로 판별한다:
- 각 이슈가 수정할 파일/영역을 예측한다 (이슈 제목, 본문, 라벨 기반)
- 같은 파일을 건드릴 가능성이 높은 이슈끼리는 같은 그룹으로 묶어 **순차 처리** 대상으로 분류
- 서로 독립적인 이슈들은 **병렬 처리** 대상으로 분류

분석 결과를 아래 형식으로 사용자에게 제안한다:

```
## 병렬 실행 제안

### 병렬 가능 (서로 독립적)
- #10: TerminalView 포커스 버그 → src/views/terminal/
- #15: 알림 배지 색상 수정 → src/components/notification/

### 순차 권장 (충돌 가능)
- #12, #14: 둘 다 grid 레이아웃 관련 → src/components/grid/

### 제외 (자동 수정 부적합)
- #8: 디자인 논의 필요 (enhancement 라벨)

병렬 가능한 #10, #15를 지금 실행할까요?
```

**사용자가 승인한 이슈들만** 다음 단계로 진행한다.

### 2. 기존 PR 중복 확인

본격적인 작업 전에, 선택된 각 이슈가 이미 관련 PR로 해결되었거나 진행 중인지 확인한다:

```bash
gh pr list --search "#{번호}" --state all --json number,title,state,mergedAt,url
```

결과를 분석하여:
- **이미 머지된 PR이 있는 이슈**: 제외하고 사용자에게 알린다 (`#{번호}는 PR #{pr번호}로 이미 해결됨`)
- **열린 PR이 있는 이슈**: 제외하고 사용자에게 알린다 (`#{번호}는 PR #{pr번호}에서 작업 진행 중`)
- **관련 PR이 없는 이슈만** 다음 단계로 진행한다

모든 이슈가 이미 해결/진행 중이면 작업을 종료하고 상태를 보고한다.

### 3. 이슈 내용 확인 (인자가 있는 경우)

각 이슈의 제목과 본문을 가져온다:
```bash
gh issue view {번호}
```

### 4. 충돌 가능성 사전 검토

이슈들의 내용을 분석하여:
- 같은 파일을 수정할 가능성이 높은 이슈 쌍이 있으면 사용자에게 경고
- 병렬 진행 여부를 확인받는다

### 5. 병렬 에이전트 실행

**각 이슈마다** 아래 조건으로 Agent를 호출한다. 모든 에이전트를 **하나의 메시지에서 동시에** 호출한다.

```
Agent(
  isolation: "worktree",
  run_in_background: true,
  prompt: 아래 템플릿
)
```

#### 에이전트 프롬프트 템플릿

```
GitHub 이슈 #{번호}를 수정한다.

## 이슈 정보
제목: {제목}
본문: {본문}

## 프로젝트 테스트 방법
- 프론트엔드: `npx vitest run` (React + TypeScript)
- 백엔드: `cargo test` (Rust / Tauri)
- 변경한 영역에 맞는 테스트 명령을 사용한다

## 작업 규칙
1. `fix/{번호}-{slug}` 브랜치를 생성한다 (slug는 이슈 제목에서 영문/숫자 요약 3단어 이내)
2. CLAUDE.md의 TDD 원칙을 따른다: 테스트를 먼저 작성하고, 구현한 뒤, 위의 테스트 명령으로 통과를 확인한다
3. ARCHITECTURE.md를 참고하여 기존 구조에 맞게 수정한다
4. 수정이 완료되면 커밋한다. 커밋 메시지에 "Fixes #{번호}" 를 포함한다
5. 리모트에 브랜치를 push한다: `git push -u origin fix/{번호}-{slug}`
6. 작업 결과를 요약하여 반환한다:
   - 브랜치명
   - 변경된 파일 목록
   - 테스트 결과
   - 주의사항 (있으면)
```

### 6. 결과 취합 및 보고

모든 에이전트가 완료되면 결과를 정리하여 보고한다:

```
## 병렬 이슈 수정 결과

| 이슈 | 브랜치 | 상태 | 변경 파일 |
|------|--------|------|-----------|
| #10  | fix/10-xxx | 완료 | 3개 |
| #12  | fix/12-yyy | 완료 | 2개 |
| #15  | fix/15-zzz | 실패 | - |

### 상세 (이슈별)
...
```

### 7. PR 생성

성공한 각 브랜치에 대해 즉시 PR을 생성한다 (사용자 확인 불필요):

```bash
gh pr create --head fix/{번호}-{slug} --base main --title "{제목}" --body "..."
```

PR 본문에는 변경 요약, 테스트 결과, `Fixes #{번호}`를 포함한다.
생성된 PR URL을 결과 보고에 포함한다.

## 주의사항

- **절대 `EnterWorktree`를 사용하지 않는다** — 메인 세션 CWD가 변경되는 것을 방지
- 메인 세션 CWD는 항상 `D:\PycharmProjects\laymux`에 유지
- 각 에이전트는 독립된 worktree에서 작업하므로 서로 간섭하지 않음
- 에이전트 완료 후 worktree는 변경사항이 있으면 유지, 없으면 자동 정리
